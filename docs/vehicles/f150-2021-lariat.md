# 2021 Ford F-150 Lariat 502A — BlueCruise

Donor firmware set from a friend's 2021 F-150 Lariat 502A with the BlueCruise package. Same platform as our [2022 F-150](f150-2022.html) set (both use `ML34` / `ML3V` prefixes), but **newer revisions**.

## Why this vehicle is interesting

Unlike the Transit, this F-150 ships with:
- **APA working fully from factory** — no speed cap hack needed.
- **LKA without the 10-second lockout.** F-150 LKA runs continuously. (Transit's 10-s lockout is a Transit-specific cal choice, not a platform limitation.)
- **But:** LKA has a **~23 mph (≈37 kph) minimum-speed gate** instead. Below that, LKA does nothing.

This is a different tuning philosophy than Transit: Transit limits LKA *duration*; F-150 limits LKA *speed range*. From a hacking standpoint, removing the F-150 min-speed gate is the next-best-thing to full-time LKA if APA full-time can't be unlocked.

## What we're looking for — 23 mph LKA minimum-speed gate

If the min-speed gate is encoded in calibration as a scalar, likely representations:

| Encoding | Value | Bytes (BE) |
|---|---|---|
| `float32` mph | 23.0 | `41 B8 00 00` |
| `float32` kph | 37.01 | `42 14 08 F5` |
| `float32` m/s | 10.28 | `41 24 7A E1` |
| `u16` raw mph | 23 | `00 17` |
| `u16` raw kph | 37 | `00 25` |
| `u16` mph ×10 | 230 | `00 E6` |
| `u16` kph ×10 | 370 | `01 72` |

**Early observation:** a byte-scan of `ML34-14D007-EDL.VBF` finds 9 occurrences each of `00 17` (u16 = 23) and `00 25` (u16 = 37), and several occur within a few bytes of each other — suggestive of threshold tables containing both mph and kph (or min and max) pairs.

Next steps:
1. **Parse the VBF properly.** The F-150 VBF uses a different block format than Transit/Escape — our `tools/vbf_decompress.py` needs a patch. After the `};` header there's a 16-byte embedded part-name followed by block data.
2. **Diff against `ML34-14D007-BDL`** (our older F-150 cal) to see what Ford changed between revisions.
3. **Locate the 23 mph threshold** in decompressed cal bytes.
4. **Correlate** with GP-relative references in `ML3V-14D003-BD.VBF` strategy disassembly to confirm which occurrence is actually the LKA min-speed gate (vs. unrelated speed thresholds for other features).

## What we've confirmed from the VBF headers

All four files:
- Use `frame_format = CAN_STANDARD` for the diagnostic channel (UDS flashing over classical CAN, even though the vehicle's control bus is CAN FD).
- Target `ecu_address = 0x730`.
- Use `data_format_identifier = 0x00` (uncompressed).

SBL is explicitly marked `sw_part_type = SBL`. Strategy is `EXE`. Cal and supplementary are `DATA`. This is the first F-150 set we have where `sw_part_type` is observable — and it **corrected an earlier misconception** in this repo that `-14D005-*` was an EPS-core partition. It is actually the Secondary Bootloader.

## Patching targets (proposed)

Once we locate the 23 mph gate:

| Goal | Patch | Risk |
|---|---|---|
| Remove LKA min-speed gate | Cal threshold → `0.0` | Low-medium. LKA will try to engage at parking speeds; PSCM or IPMA may reject. |
| Verify APA full-time | (already stock) | — |
| Enable openpilot-style continuous lateral | Combine with patched IPMA filter-out on `0x213` | Low (on the PSCM side) |

## CAN FD note

The 2021+ F-150 uses CAN FD on its chassis/ADAS bus. This matters if you're doing openpilot-style integration — a classical CAN Panda won't interoperate with the ADAS bus at full bandwidth. Use a CAN FD-capable interface (comma black panda supports CAN FD, OpenDBC has F-150 definitions).

The UDS flashing path is still classical CAN, so FORScan with a VCM-II or RLink will flash this module fine.
